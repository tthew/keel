## Notes

- **Counter-file stub pattern needs empty-string guard (iter-215 LESSON):** When writing a state-machine docker stub that progresses through values via a counter file, `: > counter` truncates to empty (not "0"); `cat ""` then yields empty string; `case "" in 0)` does NOT match (matches `*)` instead) → stub returns the default-arm value on first call, not the 0-indexed first value. Fix: `count=$(cat counter 2>/dev/null || true); count=${count:-0}` + initial `rm -f counter` (no file = default 0). Carry-forward for any future stub-based smoke that uses persistent counter state across invocations.
- **Soft-gate poll + fall-through pattern (iter-215 LESSON):** restart.sh's AI-11 fix uses a SOFT gate — on timeout, log the timeout AND proceed to start.sh regardless. This preserves forward progress under unreachable state (start.sh has its own 120s healthcheck bound), vs a hard-abort which would fail the whole restart. General rule for "wait-for-X-before-Y" gates in host-side scripts: if Y has its own downstream validation, the upstream gate can be soft. Applies to potential future poll-gates in Stories 2.7+ (Ralph auto-start readiness, claude OAuth handshake, gh auth status).
- **TTY-detect at docker-exec shim (iter-214 LESSON):** The `if [[ -t 0 ]]; then tty_flag="-it"; else tty_flag="-i"; fi` gate generalises — any host-side shim that invokes `docker exec` must TTY-detect when it wants to remain scriptable (CI / hooks / `sh -c` / `ssh host …`). Carry-forward rule for Stories 2.7+ (Ralph TUI wrapper / claude wrapper / gh wrapper / ssh wrapper): if the shim is expected to be callable from non-interactive contexts, apply the same pattern. Interactive-only shims (shell.sh, attach.sh) can keep hardcoded `-it` — interactive is the AC. **`script -qc '…' /dev/null` (util-linux) is the iteration-env-safe pty allocator** for smoking the TTY leg.
