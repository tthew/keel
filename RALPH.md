## Lessons learned

Things that went wrong, and why — so the next Ralph doesn't repeat them.

- 2026-04-19: IP can drift badly between mini-epics when the user manually swaps branches outside a Ralph iteration. First action on every iteration: reconcile `.ralph/@plan.md` against the actual branch/PR state (`git branch --show-current`, `gh pr view`) before treating the IP NOW as authoritative. Stale DONE sections are harmless; a stale NOW is misleading. Concrete reconcile for worktree-based runs: the main worktree owns `main`, so the Ralph worktree cannot `git checkout main` — use `git checkout -b <new-branch> origin/main` after `git fetch origin main`.
- 2026-04-19: Epics and architecture can disagree on scope. Architecture.md "First Implementation Priority" (lines 1395–1442) bundles Story 1.1 + devbox absorption + Ralph harness + keel-invariants seed + docs/research + all 14 package shells — far beyond Story 1.1's narrow AC in epics.md (just the monorepo + TS refs). When this happens, epics.md AC wins (it's the sprint-decomposed source of truth); record variances in the story's Project Structure Notes so dev-story knows what to defer.
- 2026-04-19: **TS5090 in per-package tsconfigs.** `pnpm -w typecheck` (Task 7) fired `TS5090: Non-relative paths are not allowed when 'baseUrl' is not set` at every per-package `tsconfig.json(3,3)` — the error is inherited-paths-flatten-into-extending-file. Root cause: `tsconfig.base.json` declared `paths` with bare values (`"packages/db/src/index.ts"`) and no `baseUrl`. TS 5.0+ accepts `paths` without `baseUrl` ONLY when each value is relative (leading `./`). Fix: prefix every `paths` value with `./`. Chose that over `baseUrl: "."` so bare-import resolution (via `moduleResolution: bundler` + package.json) stays independent of the paths map. When adding a new `@keel/<pkg>` entry to `tsconfig.base.json` in later stories, remember the `./` prefix. **Follow-up for Story 1.2 Task 1:** after moving `tsconfig.base.json` two dirs deeper into `packages/keel-invariants/`, every `paths` value needs its `./` recomputed as `../<pkg>/src/index.ts` (or `../../apps/web/src/index.ts`) — still relative, still TS5090-safe, but two path segments shift.
- 2026-04-19: **Multi-commit story PRs drift PR metadata from reality.** The first commit on `feat/story-1-1-...` was `docs(story): create Story 1.1 spec` and Ralph created PR #217 at that moment with a docs-only body. 8 task commits later, the body still said "This PR is docs-only; no code" — misleading for reviewers. Lesson: before `gh pr ready` on any multi-task story PR, re-read the existing title/body and rewrite to cover the full commit range (`git log origin/main..HEAD --oneline` is enough). The transition step in PROMPT_build.md only says "gh pr ready" — but a Draft→Open transition is the moment the PR becomes review-visible, so title/body accuracy is gate-mandatory.
- 2026-04-19: **Default `@commitlint/config-conventional` rules don't match Ralph's commit discipline (Story 1.2 Task 5).** Story spec assumed `rules: {}` would pass against Ralph's existing commit style; it doesn't. Three default rules break on Story 1.1/1.2 commits: `subject-case` (rejects sentence-case subjects like "Story 1.2 Task N — …"), `header-max-length: 100` (Ralph's story-ID-prefixed descriptive subjects routinely exceed, e.g. the spec commit at 106 chars), `body-max-line-length: 100` (file paths in commit bodies exceed and can't wrap — the Story 1.2 spec path is 130 chars). **Lesson:** when landing a config that has a pre-existing history gate (`--from origin/main --to HEAD`), always `pnpm exec commitlint` against existing history BEFORE committing the config, then layer rule overrides to match reality. Don't assume "our existing discipline satisfies defaults" without proof. **Rule-override pattern for Keel commitlint config:** disable `subject-case` + `body-max-line-length`, bump `header-max-length` to 120. If Story 1.5 (commit-msg hook) tightens rules, any new default-strictness must re-audit the in-flight branches first.
- 2026-04-19: **TS6053 when `extends` uses a workspace subpath specifier (Story 1.2 Task 1).** After moving `tsconfig.base.json` into `packages/keel-invariants/` and flipping 15 per-package `tsconfig.json` `extends` to `"@keel/keel-invariants/tsconfig"`, `pnpm -w typecheck` failed on every consumer with `TS6053: File '@keel/keel-invariants/tsconfig' not found.` Root cause: TS `extends` with a bare specifier goes through Node module resolution, which requires `node_modules/@keel/keel-invariants` to exist. pnpm only materialises workspace symlinks for packages that are **declared as dependencies**; the Story 1.1 tsconfig `paths` map is pure tsc path mapping and does NOT create symlinks. Fix: add `"@keel/keel-invariants": "workspace:*"` to `devDependencies` of every consumer (15 files: apps/web + 14 packages excluding keel-invariants itself). Story 1.2 Dev Notes line 157 assumed the symlinks existed; the story did not list this dep addition in Task 1 subtasks. **Follow-up for every future story that adds a new keel-invariants subpath-export consumer:** the consuming package.json MUST declare `@keel/keel-invariants: workspace:*` as a devDep, or the subpath won't resolve. This applies to Story 1.3 (boundary rules via eslint subpath) and any later story that extends shared configs.
- 2026-04-19: **Post-halt bookkeeping commits can orphan from `main`.** After Story 1.1's Task 8 (`96142bc`) was pushed, Ralph committed `e8a158f chore(sprint): Story 1.1 done — PR #217 Draft→Open, sprint-status + IP updated` LOCALLY but the user merged PR #217 before that commit's push. The merge fast-forwarded via `96142bc` as the second parent → `main` has Story 1.1 code but NOT the sprint-status bump to `done`. Story 1.2 iteration 1 had to correct `1-1-monorepo-scaffold-typescript-project-references: ready-for-dev` → `done` inline. **Prevention:** on the final iteration of any mini-epic, the sprint-status-update commit MUST land BEFORE the Draft→Open transition and MUST be pushed before the EPIC_DONE halt. Concretely: step 3a (IP + knowledge upkeep) should happen in the same iteration as step 5a (PR transition), so everything makes it into the PR's diff. If you halt EPIC_DONE and still have unpushed commits, those are orphaned if the user merges fast.
- 2026-04-19: **`"type": "module"` is NOT automatically in root `package.json` post-Story-1.1.** Story 1.2 Task 6 spec subtask says "Root-level `package.json` must have `"type": "module"` confirmed (it should from Story 1.1)" — it wasn't there. Root had `name`, `private`, `packageManager`, `engines`, `scripts`, `devDependencies` only. Without `"type": "module"`, the `.js` root shims (`eslint.config.js`, `prettier.config.js`, `commitlint.config.js`) get parsed as CJS by Node (default for `.js` in a package with no or `"commonjs"` type) and the `import shared from '@keel/...'` statement throws `SyntaxError: Cannot use import statement outside a module`. **Lesson:** any future story that assumes a Story 1.1 setting must VERIFY before acting — Story 1.1's Dev Notes / File List lines are the source of truth. Every package under `apps/` and `packages/` had `"type": "module"` set per-member in Story 1.1, but the ROOT package.json did not. The difference: member packages were treated as ESM-first (per-package decision); root was treated as orchestration-only (pnpm + turbo) and nobody authored JS files at root until Story 1.2 Task 6.

## Gotchas

Rough edges in tools, flaky tests, odd repo conventions, environment quirks.

- 2026-04-19: The `WIP` GitHub Marketplace app sets a commit status that stays `IN_PROGRESS` while a PR is Draft and resolves green the instant the PR is marked ready. `gh pr checks --watch --fail-fast` on a Draft PR whose only check is WIP will hang indefinitely. If IP has both "Monitor PR CI" and "Transition Draft→Open" queued and the PR is Draft + only-WIP-pending, skip Monitor and go directly to Transition. Orient step 0h ("pending check → NOW=Monitor CI") has a carve-out for this: a solitary WIP-app check on a Draft PR is not real CI.
- 2026-04-19: `.github/workflows/` does not exist yet (CI lands in Story 1.16). Until then, `gh pr view --json statusCheckRollup` returns `[]` on any PR. Monitor-PR-CI queue items added reflexively after push are vestigial — push freely, skip monitoring. Orient step 0h's "pending → Monitor" rule is a no-op with zero checks registered.
