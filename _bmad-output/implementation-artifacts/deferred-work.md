## Deferred from: code review of story-1.8 (2026-04-20)

_Story 1.8 (`invariants.manifest.ts` contract + exporter) — iter-7 `/bmad-code-review (args: "2")`. Three-layer adversarial review (Blind Hunter + Edge Case Hunter + Acceptance Auditor). Auditor returned zero AC violations; the six defers below all originate from Blind Hunter + Edge Case Hunter defensive-hardening findings that are either explicitly Story 1.9's enforcement-side scope (per Story 1.8's § Scope Carve-Out + FR43) or spec-silent tightening candidates._

- **contentHash drift validation at manifest-use time.** _Source:_ Blind Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts` (all 9 entries). _Nature:_ the manifest stores a pre-computed `contentHash` but never re-reads the `sourcePath` file to verify the hash is still current. _Carry-to:_ Story 1.9 sync-gate — FR43 enforcement side explicitly owns the anchor-walker + file-read + sha256 re-computation + drift-detection flow. Absorbed naturally; no action needed ahead of Story 1.9.
- **sourcePath path-traversal / format guard in Zod schema.** _Source:_ Blind Hunter + Edge Case Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts:6` (`sourcePath: z.string().min(1)`). _Nature:_ schema accepts absolute paths, `..` traversal, Windows backslashes, leading slashes. Current 9 entries are all repo-relative forward-slash, so no live exposure. _Carry-to:_ Story 1.9 walker does file-resolution and will naturally reject non-resolvable paths; if tightening the schema itself is wanted earlier, a one-line `.regex(/^[^/\\][^\\]*$/).refine(p => !p.includes('..'))` does it — but that's scope creep vs Story 1.8's contract-only remit.
- **`id` uniqueness across array not enforced by schema.** _Source:_ Blind Hunter + Edge Case Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts:11` (`InvariantsSchema = z.array(InvariantSchema)`). _Nature:_ Zod's `z.array` permits duplicate elements; two entries with the same `id` would silently validate. Acceptance Auditor confirmed current 9 IDs are unique. _Carry-to:_ Story 1.9's drift-detection gate treats unregistered / duplicate-registered rules as drift per AC 3 enforcement side — duplicates surface there. Alternatively, a `.superRefine(arr => Set uniqueness check)` on `InvariantsSchema` in Story 1.9's refactor pass.
- **Shared-sourcePath cross-entry `contentHash` consistency refine.** _Source:_ Edge Case Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts:11`. _Nature:_ four current entries pair up on two shared source files (`eslint.config.keel-invariants.js` hosts `INV-eslint-shared` + `INV-eslint-import-boundary`; `.pre-commit-config.yaml` hosts `INV-prek-pre-commit-config` + `INV-prek-commit-msg-config`). Whole-file carve-out means these pairs MUST share identical `contentHash` values (they do — auditor verified). If a future authoring error broke that invariant (same `sourcePath`, different `contentHash`), the schema wouldn't catch it. _Carry-to:_ Story 1.9 walker re-computes `sha256` per distinct `sourcePath` and surfaces cross-entry mismatch immediately.
- **`invariants` array immutability — `readonly` / `Object.freeze`.** _Source:_ Blind Hunter + Edge Case Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts:89` (`export const invariants: Invariant[]`). _Nature:_ consumer of `@keel/keel-invariants` receives a live reference to a mutable array; a misbehaving consumer (`invariants.push(...)` / `invariants[0].id = '...'`) would corrupt the module-level singleton for every other importer. No current consumer is known to mutate it. _Carry-to:_ Story 1.9 refactor — two-line change (`readonly Invariant[]` type + `Object.freeze(InvariantsSchema.parse(raw))`). Spec-silent hardening; low-cost but out-of-scope for contract-only 1.8.
- **Schema-evolution metadata on `Invariant` entries (`schemaVersion` / `deprecated` / `since`).** _Source:_ Blind Hunter. _Location:_ `packages/keel-invariants/src/invariants.manifest.ts:3-9` (`InvariantSchema` object shape). _Nature:_ no first-class way to mark an invariant as deprecated-but-not-yet-drifted, no schema version field on the root manifest, no `since` / introduced-in-story field per entry. _Carry-to:_ Story 1.9's enforcement side will need "deprecated ≠ drift" semantics at minimum (otherwise sunsetting a rule and removing its registration both look like drift). Natural home for this expansion. If earlier would be valuable, a later substrate-hardening story (1.16 scope or adjacent) can add it.

### Classification methodology (`/bmad-code-review (args: "2")` — "Create action items")

Three parallel adversarial layers per `.claude/skills/bmad-code-review/steps/step-02-review.md`: Blind Hunter (`bmad-review-adversarial-general`; diff-only, no project context), Edge Case Hunter (`bmad-review-edge-case-hunter`; diff + repo read access; path-enumeration method), Acceptance Auditor (diff + spec file + context docs; AC-violation-only).

Triage bucketing per `step-03-triage.md`:

- `patch` — fix unambiguous, no human input needed.
- `defer` — real issue, not actionable now (e.g. scope carve-out, spec-silent hardening).
- `dismiss` — false positive, by-design, or handled elsewhere.
- `decision_needed` — ambiguous choice (not applicable when auditor found zero AC violations).

Under adversarial-triage-is-default (per `.ralph/PROMPT_build.md § Story Lifecycle: "EVERY CR action item becomes a QUEUE fix task unless IP records defer: <reason> per item. Adversarial triage is the default."`): every non-dismissed finding must either get a `patch` fix task queued or an explicit `defer: <reason>` recorded. Story 1.8's 6 defers above are all either Story 1.9-enforcement-side scope or spec-silent hardening — no patches needed for the `sm-verified → done` transition.
