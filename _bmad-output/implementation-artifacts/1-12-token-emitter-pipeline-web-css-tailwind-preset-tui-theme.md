## Story

As a front-end developer or Ralph (TUI consumer),
I want `packages/ui/scripts/generate-tokens.ts` that reads `packages/ui/tokens.json` (Story 1.11 source) and deterministically emits three artefacts — `packages/ui/src/tokens.css` (web CSS custom properties), `packages/ui/tailwind.preset.ts` (Tailwind v4 preset with `theme.extend`), `packages/devbox/tui/theme.py` (Textual Python theme constants),
So that every downstream consumer (Epic 7 `@keel/ui` primitives + catalog; Epic 12 shape-aware templates; Epic 3 Story 3.33 default TUI chrome) reads from one source with byte-identical re-emission guarantees and the pipeline passes Epic 13's reproducibility CI check (UX-DR3, UX-DR6; PRD F2; FR67 purity contract adapted).

## Acceptance Criteria

   **When** I run `pnpm --filter @keel/ui generate-tokens`,
   **Then** three output files are emitted at the paths below (NEW or OVERWRITTEN, never appended to):
   - `packages/ui/src/tokens.css` — web CSS custom properties inside `:root { }` (base = light) + `[data-theme="dark"] { }` (dark overlay) blocks.
   - `packages/ui/tailwind.preset.ts` — Tailwind v4-compatible preset: a named export `keelTailwindPreset` conforming to Tailwind's `Config` shape (subset — see § AC 4), with `theme.extend.{colors,spacing,fontSize,fontFamily,borderRadius,transitionDuration,screens}` populated from tokens.json.
   - `packages/devbox/tui/theme.py` — Python module with a `Theme` dataclass (or named tuples / frozen namespace — dev picks; see § AC 5) exposing nested namespaces `theme.colors.{surface,text,border,accent,status,severity,state,neutral}` + `theme.motion` + `theme.density` keyed by semantic snake_case names.
   **And** every output file begins with a provenance header comment matching the exact format in § AC 1 carve-out (provenance header shape): source-file name + source-file commit SHA resolved at emit time (git short SHA of the file's latest commit — 8 or 12 hex chars) + emitter version string (pinned in the emitter source as a `const EMITTER_VERSION = '1.0.0'` literal) + DO-NOT-EDIT warning line.
   **And** the emitter reads only `packages/ui/tokens.json` (+ the Story 1.10 schema for optional validation — see § AC 6 carve-out) — it does not read `docs/invariants/tokens.md` or any other file beyond the three written outputs.

   **Story 1.12 scope carve-out — emitter location + output paths (ratifies architecture.md:916-917 / :1217-1218 variance; parallels Story 1.11 Direction A precedence rule).** Three sources name different emitter paths: (a) **Story 1.12 AC text at `epics.md:985`** names `packages/ui/scripts/generate-tokens.ts` + `packages/ui/src/tokens.css` + `packages/ui/tailwind.preset.ts` + `packages/devbox/tui/theme.py`; (b) **UX-DR3 at `epics.md:367`** names the same exact paths verbatim; (c) **architecture.md `§ Cross-cutting — Design System (F1 + F2)` at lines 1216-1218** names `packages/keel-invariants/src/emit-tailwind-theme.ts` + `emit-textual-theme.ts` emitting to `apps/web/tailwind.config.ts` + "Ralph TUI theme" (path unspecified). The variance is reconciled by **Story 1.12 AC text wins** per the Story 1.11 carve-out precedent pattern (epic-AC-text > architecture when they disagree — epic-AC is the normative spec consumed by `/bmad-create-story`). Architecture.md:916-917 + :1217-1218 reflects an earlier design that pre-dated the W1 party-mode amendment (`epics.md:653-666`) which relocated the tokens substrate from `packages/keel-invariants/` into `packages/ui/`. Story 1.11 § Dev Notes:165 forbids changes to `packages/keel-invariants/` for this class of work. Document the path ratification in the emitter source-file header block.

   **Story 1.12 scope carve-out — provenance header shape.** Exact prescribed form (all three outputs, format-adjusted per target language comment syntax):

   ```
   // AUTOGENERATED from packages/ui/tokens.json — DO NOT EDIT.
   // Source file commit SHA: <short-sha>
   // Emitter: packages/ui/scripts/generate-tokens.ts @ v<EMITTER_VERSION>
   // Regenerate via: pnpm --filter @keel/ui generate-tokens
   ```

   - CSS variant uses `/* ... */` delimiters (CSS supports `//` since Tailwind v4, but `/* */` is safer cross-tool).
   - Tailwind preset (TS) uses `// ...` line comments (TypeScript native).
   - Python variant uses `# ...` line comments (Python native).
   - The `<short-sha>` is resolved at emit time via `git rev-parse --short=12 HEAD -- packages/ui/tokens.json` (the latest commit that touched the source file — NOT `HEAD` of the repo, which drifts as soon as the emitter commits the outputs; NOT a hash of the file content, which would collide with Story 1.13's sync-gate hash). If the emitter is run on an uncommitted state (e.g. during dev-story itself, before the Story 1.12 commit lands), substitute `uncommitted-<content-sha256-16>` where `<content-sha256-16>` is the first 16 chex of `sha256(tokens.json)` — preserves determinism modulo source content without requiring git state. Capture the exact resolver invocation in § Dev Agent Record → Debug Log.
   - **EMITTER_VERSION** is a literal constant in `generate-tokens.ts` (e.g. `export const EMITTER_VERSION = '1.0.0';`). Bumping the version is a semver decision owned by future emitter-evolution stories; Story 1.12 ships `1.0.0`. The emitter version rides alongside the source-file SHA so byte-identical outputs at two different emitter versions still diff — protecting against silent algorithm drift across Story 1.13 source-output-sync gate re-runs.

   **Story 1.12 scope carve-out — the emitter is a pure producer; it does NOT mutate `packages/ui/tokens.json` or `packages/ui/tokens.schema.json` (Story 1.10 + 1.11 frozen sources).** Any attempt to consume the rationale doc (`docs/invariants/tokens.md`) is also out of scope — the rationale doc is for humans + future Story-7 catalog prose, not the emitter.

2. **Given** the emitter is deterministic (FR67-adapted purity contract: same input → same output, no time-dependent values, no RNG, no filesystem reads beyond `packages/ui/tokens.json`, no network),
   **When** I run `pnpm --filter @keel/ui generate-tokens` twice back-to-back with no source changes,
   **Then** both runs produce byte-identical outputs at all three target paths (verified by `sha256sum` comparison)
   **And** `git diff packages/ui/src/tokens.css packages/ui/tailwind.preset.ts packages/devbox/tui/theme.py` after the second run is empty.

   **Story 1.12 scope carve-out — determinism substrate-evidence is TWO smokes, not three.** The `git diff` clean-path check is smoke 1. Smoke 2 is the byte-identical `sha256sum` round-trip: capture hashes pre-run-2 and post-run-2; require equality. Both smokes run at § Task 6 substrate-verification; evidence lands in § Completion Notes per Story 1.10/1.11 Task-4 patterns. The third determinism smoke — **running the emitter against a known-bad source (e.g. a tokens.json with an unresolved `{alias}` pointing at a missing leaf) must fail loudly with a structured error referencing the offending path** — is a NEGATIVE-space smoke that lands at Story 1.13's schema-validation gate time (not Story 1.12). Story 1.12's emitter may optionally perform defensive alias-resolution checks, but the formal pre-commit "reject malformed tokens.json" gate is Story 1.13 scope (per epics.md:1028 — "the pre-commit hook validates it against `tokens.schema.json`").

   **Story 1.12 scope carve-out — the emit-time SHA resolver accepts a non-committed state.** Per § AC 1 provenance-header carve-out, when `git rev-parse --short=12 HEAD -- packages/ui/tokens.json` would return empty (file is newly-added, uncommitted), the emitter substitutes `uncommitted-<content-sha256-16>`. This means the emitter is deterministic even in uncommitted states: the same uncommitted content-hash pairs with the same fallback-SHA. This preserves the byte-identical-round-trip invariant in all cases.

3. **Given** the emitted `packages/ui/src/tokens.css`,
   **When** consumed in a web app (Epic 7 Story 7-1 `packages/ui/src/index.ts` imports + re-exports it; apps/web roots import via `@keel/ui/tokens.css`),
   **Then** every semantic token from `packages/ui/tokens.json` is available as a CSS custom property under `:root { }` (light mode base) using **kebab-case** CSS var names following the `--<category>-<semantic-name>-<modifier?>` convention matching architecture.md:693's Design-token ID pattern (colons in DTCG aliases `{color.surface.raised}` map to `var(--color-surface-raised)`; dots become kebab segments)
   **And** every `$modes.dark` overlay re-defines the same CSS vars under `[data-theme="dark"] { }` (scoped per architecture.md + ux-design-specification.md:484 "generator emits both to CSS variables (`:root` and `[data-theme=\"dark\"]`)")
   **And** all 8 token groups are emitted: `color.*`, `type.*`, `font.*`, `space.*`, `radius.*`, `motion.*`, `density.*`, `breakpoint.*` — no group is omitted.

   **Story 1.12 scope carve-out — CSS var naming convention.** DTCG aliases in `tokens.json` use dot-path syntax (`{color.surface.raised}`). The emitter converts dot-paths to kebab-case CSS var names via the rule `path.split('.').join('-')`, prefixed with `--`: `color.surface.raised` → `--color-surface-raised`; `motion.scale` → `--motion-scale`; `status.info.fg` → `--status-info-fg` (wait — this is four segments with `.` but the DTCG path is actually `color.status.info.fg` — verify at authoring time that the emitter uses the FULL token-source path, not a flattened semantic alias). Known concrete pairs (NON-EXHAUSTIVE — dev enumerates exhaustively from tokens.json leaf walk):
   - `color.surface.raised` → `var(--color-surface-raised)`
   - `color.status.success.fg` → `var(--color-status-success-fg)`
   - `color.accent.500` → `var(--color-accent-500)`
   - `color.severity.critical` → `var(--color-severity-critical)` (note this resolves to `{color.status.critical.fg}` in source; the emitter emits BOTH the source alias and a computed `var()` chain OR flattens to the resolved leaf value at emit time — **pick flatten-at-emit** per § AC 3 scope carve-out "alias-resolution strategy" below)
   - `motion.swift` → `var(--motion-swift)`
   - `density.compact` → `var(--density-compact)`

   **Story 1.12 scope carve-out — alias-resolution strategy (FLATTEN AT EMIT TIME, not `var()` chain).** DTCG aliases `{color.accent.500}` resolve at the source-authoring layer; the emitter walks the alias graph at emit time and writes the **resolved leaf value** (OKLCH literal, `px` literal, etc.) into every CSS var. This means `--color-severity-critical` emits `oklch(45% 0.20 15)` directly (Story 1.11's `color.status.critical.fg` leaf value), NOT `var(--color-status-critical-fg)`. Rationale: (a) keeps downstream CSS readable (no chained `var()` lookups); (b) decouples semantic-alias mutations at source time from consumer-side cascade complexity; (c) matches DTCG tooling convention (Style Dictionary, Tokens Studio emit flattened values by default); (d) simplifies the Tailwind preset emitter — no second-pass resolution needed. The emitter detects cycles (`{A} → {B} → {A}`) via standard DFS + visited-set and fails loudly on cycle detection with a structured error naming the cycle edges. No cycles exist in the Story 1.11 tokens.json (iter-58 SM-verified); the cycle detector is a defensive guard for future source evolution.

   **Story 1.12 scope carve-out — the emitter does NOT resolve `motion.scale * motion.swift` product at source time.** Per Story 1.11 AC 3 carve-out: the dial (`motion.scale = 1`) × tier (`motion.swift = 200ms`) product is Story 1.12's runtime-layer responsibility. But "runtime-layer" here means **consumer-layer**, not emitter-time. The emitter emits the tier as a `200ms` CSS var AND the dial as a unitless `1` CSS var; consumer code (Epic 7 primitives, Epic 3 Story 3.33 TUI) multiplies at runtime via `calc(var(--motion-swift) * var(--motion-scale))` or Python equivalent. Story 1.12 emits both; the product computation does NOT land in the CSS emitter.

4. **Given** the emitted `packages/ui/tailwind.preset.ts`,
   **When** imported into a Tailwind v4 config (Epic 7 Story 7-1 imports `@keel/ui/tailwind.preset` so consumer apps/web extends the Keel token preset),
   **Then** the file exports a named constant `keelTailwindPreset` of type-shape compatible with Tailwind's `Config['theme']['extend']` family
   **And** every semantic color token is exposed under `theme.extend.colors.<kebab-slug>` (e.g. `theme.colors['surface-raised']`, `theme.colors['status-success']`, `theme.colors['status-success-fg']`, `theme.colors['accent-500']`)
   **And** non-color groups flow into their matching Tailwind namespaces: `type.*` → `theme.extend.fontSize.*`, `space.*` → `theme.extend.spacing.*`, `radius.*` → `theme.extend.borderRadius.*`, `motion.*` → `theme.extend.transitionDuration.*` (durations only; motion.scale dial emits under a custom `theme.extend.motionScale.*` key), `breakpoint.*` → `theme.extend.screens.*`, `font.*` → `theme.extend.fontFamily.*`.

   **Story 1.12 scope carve-out — Tailwind flat-naming convention (no nested objects under `colors`).** Tailwind's `theme.extend.colors` supports both flat string keys (`'surface-raised': 'oklch(...)'`) and nested objects (`surface: { raised: 'oklch(...)' }`). The emitter MUST emit **flat string keys** with kebab-slugs matching the CSS-var naming in § AC 3: `'surface-raised'`, `'status-success-fg'`, `'accent-500'`. Rationale: (a) matches Tailwind class-name generation (`bg-surface-raised`, `text-status-success-fg`, `border-accent-500`) per ux-design-specification.md:154 "cross-runtime semantic tokens"; (b) avoids the nested-object trap where `bg-surface-raised` wouldn't resolve without Tailwind's `extend` merging rules doing the flatten; (c) makes the Tailwind preset shape identical to the Python dict shape in AC 5, reducing cross-target drift.

   **Story 1.12 scope carve-out — Tailwind v4 preset shape.** Tailwind v4's preset export uses the `PartialConfig` type from `tailwindcss` (ESM import: `import type { Config } from 'tailwindcss';`). The emitter writes a minimal shape matching `Partial<Config>['theme']['extend']` without requiring Tailwind as a runtime dep at emit time — the emitter can write the TS AST literally (`export const keelTailwindPreset = { theme: { extend: { colors: { ... } } } } as const;`) without importing Tailwind types. Consumer apps (Epic 7, apps/web) import Tailwind types at consumer-build time. This means **Tailwind is NOT a devDependency of `@keel/ui` at Story 1.12 time** (consistent with "expect minimal new deps" — see § Task 1 devDep posture). The preset is type-compatible by shape; Tailwind-import happens at consumer-build.

   **Story 1.12 scope carve-out — color values land as OKLCH literals (flattened from aliases per § AC 3), NOT as `var(--color-*)` references.** Tailwind preset is consumed at build-time (compile-time); the preset MUST carry literal values so Tailwind's JIT compiler generates the matching CSS classes at consumer build. If the preset used `var()` references, the JIT'd output would be `.bg-surface-raised { background-color: var(--color-surface-raised); }` — which works IF the CSS var is also emitted (which it is, via `tokens.css`) BUT duplicates the source value definition across two files and creates a second point of failure at source-output-sync gate time (Story 1.13). The simpler + idiomatic pattern: Tailwind preset carries literals, `tokens.css` carries the same literals under CSS vars. Consumer apps import BOTH — Tailwind for class generation, `tokens.css` for runtime theme switching (`[data-theme="dark"]` cascade).

5. **Given** the emitted `packages/devbox/tui/theme.py`,
   **When** imported into a Textual app (Epic 3 Story 3.33 default TUI chrome imports it via `from packages.devbox.tui.theme import theme` or Python-package-relative import once `packages/devbox/tui/__init__.py` lands),
   **Then** Python constants map 1:1 to semantic tokens under the namespace `theme.colors.<snake_case_slug>` (e.g. `theme.colors.surface_raised`, `theme.colors.status_success_fg`, `theme.colors.accent_500`)
   **And** the file carries enough of a Python-package shape to be import-resolvable: either a module-level `theme` object (preferred — avoids the dataclass-vs-named-tuple-vs-namespace choice at Story 1.12 time) with attribute access, OR an explicitly-documented alternative in § Dev Agent Record → Completion Notes explaining the chosen shape
   **And** the same 8 token groups that Tailwind exposes are present in the Python theme (minus `breakpoint.*` — TUI doesn't use responsive breakpoints; see carve-out below).

   **Story 1.12 scope carve-out — TUI theme.py shape + `breakpoint.*` omission.** Textual (the Python TUI framework) does not use responsive breakpoints — TUI layout is cell-based and always fits the terminal dimensions per viewport query at runtime. The emitter MAY omit `breakpoint.*` entirely from `theme.py` OR emit it as a dead-weight attribute for consistency. **Pick: emit `theme.breakpoint.{sm,md,lg,xl,xl2}` as `int` constants (px values as plain numbers; `xl2` substitutes `2xl` since Python identifiers can't start with a digit)** so the shape matches Tailwind + CSS parity — zero functional TUI consumption, but preserves stable-ID invariance for cross-runtime consistency per architecture.md:90 ("shared status / severity / density vocabulary consumed by both the Textual TUI theme AND the Tailwind config").

   **Story 1.12 scope carve-out — OKLCH → hex or OKLCH-pass-through for TUI.** Textual supports CSS-style color values including `oklch(...)` as of Textual ≥ 0.80 (2024-Q3 release). If the dev agent targets a Textual version that supports OKLCH natively (confirm at `/bmad-dev-story` via `pip show textual` or the Story 3.33 spec pin), emit OKLCH string literals directly (`status_success_fg: str = "oklch(50% 0.15 145)"`). If the target Textual version is older or OKLCH-unsupported, emit sRGB hex fallbacks computed via a deterministic OKLCH→sRGB conversion (no library dependency — inline the conversion function in the emitter). **Default: emit OKLCH literals as strings** (let the consumer Textual parse them). Capture the chosen format + Textual version pin in § Dev Agent Record → Completion Notes + a Debug Log entry. This choice can be re-litigated at Story 3.33 time when Epic 3 TUI consumption lands.

   **Story 1.12 scope carve-out — Python 3 and syntax posture.** Target Python 3.10+ (PEP 604 `int | None` union syntax; `match/case` available). No opinion on dataclass vs SimpleNamespace vs frozen named-tuple — dev picks one and documents it; the shape-preservation concern (AC 5 "Python constants map 1:1") takes precedence over aesthetic-tool choice. Do NOT use runtime imports of third-party libraries (no `numpy`, no `colour`, no `pydantic`) — the emitted `theme.py` must be import-resolvable in the minimal Python stdlib. This is a downstream FR67-purity analogue: the emitter is pure (AC 6), AND its output is consumer-runtime-pure (no transitive deps added by virtue of importing `theme.py`).

6. **Given** the emitter is pure per adapted FR67 contract (`epics.md:1013-1016`),
   **When** invoked (any of: `pnpm --filter @keel/ui generate-tokens`, direct `tsx scripts/generate-tokens.ts`, future CI re-run under Epic 13's reproducibility smoke),
   **Then** the emitter: (a) opens zero network connections (no `fetch`, no `https.request`, no DNS lookup); (b) reads exactly one filesystem path — `packages/ui/tokens.json` — and optionally `packages/ui/tokens.schema.json` for defensive validation (OPTIONAL per § AC 6 carve-out); (c) writes exactly the three output paths from § AC 1, no other filesystem effects (no temp files, no `.cache/` writes, no `git` mutations, no `npm` side effects); (d) uses no time-dependent values (no `Date.now()`, no `new Date()`, no `process.hrtime()`) — the provenance header's `<short-sha>` is the ONLY time-adjacent value and is resolved deterministically from git state, not wall clock; (e) uses no RNG; (f) uses no env vars beyond those Node needs to start (no `process.env.KEEL_*` reads in the emitter source)
   **And** passes Epic 13's reproducibility CI check when that check lands (Story 13-1 / 13-2; forward-reference only at 1.0 — the check itself is outside Story 1.12 scope).

   **Story 1.12 scope carve-out — Epic 13 forward-reference is aspirational + non-blocking.** Epic 13's reproducibility CI check does not exist at Story 1.12 time (Epic 13 is 13 epics away from this story in the roadmap; sprint-status shows all 14 Epic-13 stories backlog). The AC text "passes Epic 13's reproducibility CI check" is a forward-contract statement meaning the emitter must BE reproducible in the FR67-analogue-pure sense — it does not require the CI check itself to be live. When Epic 13 Stories 13-1 + 13-2 land, they'll add a content-hash-diff step to `pre-merge-fast` that runs the emitter and diffs output against committed output; the Story 1.12 emitter must survive that check on landing day. Story 1.12 substrate-verifies purity via the § Task 6 determinism smoke (byte-identical round-trip); Epic 13 adds the CI enforcement layer.

   **Story 1.12 scope carve-out — OPTIONAL defensive schema validation at emit time.** The emitter MAY optionally validate `packages/ui/tokens.json` against `packages/ui/tokens.schema.json` as a defensive pre-emit check. This is explicitly OPTIONAL because: (a) Story 1.13 owns the pre-commit schema-validation gate — making it load-bearing in the emitter duplicates Story 1.13's scope; (b) the emitter reading the schema adds a second filesystem path to the pure-contract (AC 6 allows this as an explicit opt-in — two paths is fine, unbounded paths is not); (c) during normal operation tokens.json is already schema-valid (Story 1.11 iter-58 SM-verified + iter-59 CR-verified), so the emit-time check is belt-and-suspenders. **Default: skip the schema check** — the emitter trusts tokens.json's shape. Dev may add the check at implementation time if it helps debug alias-resolution edge cases; if added, capture the decision + rationale in § Dev Agent Record → Completion Notes. Story 1.13's pre-commit schema gate will catch any real drift before it reaches the emitter.

   **Story 1.12 scope carve-out — gamut-mapping is OPT-OUT at Story 1.12 time.** Story 1.11 iter-59 CR surfaced five OKLCH source values with slightly-negative linear-RGB components (`status.warning.fg` hue 75, `status.error.fg` hue 25, `status.critical.fg` hue 15 — all within ~1% of the sRGB gamut boundary). Browsers clip these on render, altering perceived hue/chroma. The emitter MAY implement a gamut-check + fallback-to-in-gamut-neighbour pass on CSS emission (per `deferred-work.md § Deferred from: code review of story-1.11` defer #5 — "Story 1.12 emitter — optional gamut-check + fallback-to-in-gamut-neighbour pass on emit"). **Default: skip gamut-mapping** — emit OKLCH literals as-is and accept browser clipping; the WCAG-AA contrast gate is Story 1.13 scope and operates on emitted/rendered values anyway. If the dev chooses to implement gamut-mapping, document the algorithm choice + affected leaves + before/after values in § Dev Agent Record → Debug Log. Surface-slot shadow-machinery (defer #4) is entirely deferred to a follow-up story (Story 1.13 + new Story-1.17-TBD) — Story 1.12 does NOT author shadow tokens, does NOT render surface differentiation, does NOT alter the 5pp dark-mode surface.inset L-delta. Shadow-machinery requires mutating `tokens.json` which is Story 1.11-frozen — a follow-up story must land shadow tokens in source before Story 1.12-style emission can render them.
