## Story

3. **Given** the emitted `packages/ui/src/tokens.css`,
   **When** consumed in a web app (Epic 7 Story 7-1 `packages/ui/src/index.ts` imports + re-exports it; apps/web roots import via `@keel/ui/tokens.css`),
   **Then** every semantic token from `packages/ui/tokens.json` is available as a CSS custom property under `:root { }` (light mode base) using **kebab-case** CSS var names following the `--<category>-<semantic-name>-<modifier?>` convention matching architecture.md:693's Design-token ID pattern (colons in DTCG aliases `{color.surface.raised}` map to `var(--color-surface-raised)`; dots become kebab segments)

   **And** all 8 token groups are emitted: `color.*`, `type.*`, `font.*`, `space.*`, `radius.*`, `motion.*`, `density.*`, `breakpoint.*` ‚Äî no group is omitted.

   **Story 1.12 scope carve-out ‚Äî CSS var naming convention.** DTCG aliases in `tokens.json` use dot-path syntax (`{color.surface.raised}`). The emitter converts dot-paths to kebab-case CSS var names via the rule `path.split('.').join('-')`, prefixed with `--`: `color.surface.raised` ‚Üí `--color-surface-raised`; `motion.scale` ‚Üí `--motion-scale`; `status.info.fg` ‚Üí `--status-info-fg` (wait ‚Äî this is four segments with `.` but the DTCG path is actually `color.status.info.fg` ‚Äî verify at authoring time that the emitter uses the FULL token-source path, not a flattened semantic alias). Known concrete pairs (NON-EXHAUSTIVE ‚Äî dev enumerates exhaustively from tokens.json leaf walk):
   - `color.surface.raised` ‚Üí `var(--color-surface-raised)`
   - `color.status.success.fg` ‚Üí `var(--color-status-success-fg)`
   - `color.accent.500` ‚Üí `var(--color-accent-500)`
   - `color.severity.critical` ‚Üí `var(--color-severity-critical)` (note this resolves to `{color.status.critical.fg}` in source; the emitter emits BOTH the source alias and a computed `var()` chain OR flattens to the resolved leaf value at emit time ‚Äî **pick flatten-at-emit** per ¬ß AC 3 scope carve-out "alias-resolution strategy" below)
   - `motion.swift` ‚Üí `var(--motion-swift)`
   - `density.compact` ‚Üí `var(--density-compact)`

   **Story 1.12 scope carve-out ‚Äî alias-resolution strategy (FLATTEN AT EMIT TIME, not `var()` chain).** DTCG aliases `{color.accent.500}` resolve at the source-authoring layer; the emitter walks the alias graph at emit time and writes the **resolved leaf value** (OKLCH literal, `px` literal, etc.) into every CSS var. This means `--color-severity-critical` emits `oklch(45% 0.20 15)` directly (Story 1.11's `color.status.critical.fg` leaf value), NOT `var(--color-status-critical-fg)`. Rationale: (a) keeps downstream CSS readable (no chained `var()` lookups); (b) decouples semantic-alias mutations at source time from consumer-side cascade complexity; (c) matches DTCG tooling convention (Style Dictionary, Tokens Studio emit flattened values by default); (d) simplifies the Tailwind preset emitter ‚Äî no second-pass resolution needed. The emitter detects cycles (`{A} ‚Üí {B} ‚Üí {A}`) via standard DFS + visited-set and fails loudly on cycle detection with a structured error naming the cycle edges. No cycles exist in the Story 1.11 tokens.json (iter-58 SM-verified); the cycle detector is a defensive guard for future source evolution.

   **Story 1.12 scope carve-out ‚Äî the emitter does NOT resolve `motion.scale * motion.swift` product at source time.** Per Story 1.11 AC 3 carve-out: the dial (`motion.scale = 1`) √ó tier (`motion.swift = 200ms`) product is Story 1.12's runtime-layer responsibility. But "runtime-layer" here means **consumer-layer**, not emitter-time. The emitter emits the tier as a `200ms` CSS var AND the dial as a unitless `1` CSS var; consumer code (Epic 7 primitives, Epic 3 Story 3.33 TUI) multiplies at runtime via `calc(var(--motion-swift) * var(--motion-scale))` or Python equivalent. Story 1.12 emits both; the product computation does NOT land in the CSS emitter.

4. **Given** the emitted `packages/ui/tailwind.preset.ts`,
   **When** imported into a Tailwind v4 config (Epic 7 Story 7-1 imports `@keel/ui/tailwind.preset` so consumer apps/web extends the Keel token preset),
   **Then** the file exports a named constant `keelTailwindPreset` of type-shape compatible with Tailwind's `Config['theme']['extend']` family
   **And** every semantic color token is exposed under `theme.extend.colors.<kebab-slug>` (e.g. `theme.colors['surface-raised']`, `theme.colors['status-success']`, `theme.colors['status-success-fg']`, `theme.colors['accent-500']`)
   **And** non-color groups flow into their matching Tailwind namespaces: `type.*` ‚Üí `theme.extend.fontSize.*`, `space.*` ‚Üí `theme.extend.spacing.*`, `radius.*` ‚Üí `theme.extend.borderRadius.*`, `motion.*` ‚Üí `theme.extend.transitionDuration.*` (durations only; motion.scale dial emits under a custom `theme.extend.motionScale.*` key), `breakpoint.*` ‚Üí `theme.extend.screens.*`, `font.*` ‚Üí `theme.extend.fontFamily.*`.

   **Story 1.12 scope carve-out ‚Äî Tailwind flat-naming convention (no nested objects under `colors`).** Tailwind's `theme.extend.colors` supports both flat string keys (`'surface-raised': 'oklch(...)'`) and nested objects (`surface: { raised: 'oklch(...)' }`). The emitter MUST emit **flat string keys** with kebab-slugs matching the CSS-var naming in ¬ß AC 3: `'surface-raised'`, `'status-success-fg'`, `'accent-500'`. Rationale: (a) matches Tailwind class-name generation (`bg-surface-raised`, `text-status-success-fg`, `border-accent-500`) per ux-design-specification.md:154 "cross-runtime semantic tokens"; (b) avoids the nested-object trap where `bg-surface-raised` wouldn't resolve without Tailwind's `extend` merging rules doing the flatten; (c) makes the Tailwind preset shape identical to the Python dict shape in AC 5, reducing cross-target drift.

   **Story 1.12 scope carve-out ‚Äî Tailwind v4 preset shape.** Tailwind v4's preset export uses the `PartialConfig` type from `tailwindcss` (ESM import: `import type { Config } from 'tailwindcss';`). The emitter writes a minimal shape matching `Partial<Config>['theme']['extend']` without requiring Tailwind as a runtime dep at emit time ‚Äî the emitter can write the TS AST literally (`export const keelTailwindPreset = { theme: { extend: { colors: { ... } } } } as const;`) without importing Tailwind types. Consumer apps (Epic 7, apps/web) import Tailwind types at consumer-build time. This means **Tailwind is NOT a devDependency of `@keel/ui` at Story 1.12 time** (consistent with "expect minimal new deps" ‚Äî see ¬ß Task 1 devDep posture). The preset is type-compatible by shape; Tailwind-import happens at consumer-build.

   **Story 1.12 scope carve-out ‚Äî color values land as OKLCH literals (flattened from aliases per ¬ß AC 3), NOT as `var(--color-*)` references.** Tailwind preset is consumed at build-time (compile-time); the preset MUST carry literal values so Tailwind's JIT compiler generates the matching CSS classes at consumer build. If the preset used `var()` references, the JIT'd output would be `.bg-surface-raised { background-color: var(--color-surface-raised); }` ‚Äî which works IF the CSS var is also emitted (which it is, via `tokens.css`) BUT duplicates the source value definition across two files and creates a second point of failure at source-output-sync gate time (Story 1.13). The simpler + idiomatic pattern: Tailwind preset carries literals, `tokens.css` carries the same literals under CSS vars. Consumer apps import BOTH ‚Äî Tailwind for class generation, `tokens.css` for runtime theme switching (`[data-theme="dark"]` cascade).
