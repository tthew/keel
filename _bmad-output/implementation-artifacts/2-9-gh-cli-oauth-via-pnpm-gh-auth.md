# Story 2.9: `gh` CLI OAuth via `pnpm gh:auth`

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a fork operator,
I want `pnpm gh:auth` to trigger the `gh auth login` browser flow surfaced to my host terminal, with tokens persisted in the named Docker volume at `/home/dev/.config/gh/`,
So that Ralph can push commits and read PR state without per-iteration re-auth (FR3 gh side).

## Acceptance Criteria

1. **OAuth flow surfaced to host terminal.** Given a fresh devbox, when I run `pnpm gh:auth`, then the command invokes `gh auth login` inside the container and the OAuth URL is surfaced to my host terminal and completing the flow in a host browser returns control to the CLI.

2. **Token persisted in named volume.** Given the flow completes, when tokens are stored, then they persist at `/home/dev/.config/gh/` inside the named volume from Story 2.5 and no host bind-mount is involved.

3. **Token survives restart.** Given a subsequent devbox restart (`pnpm devbox:restart` or `pnpm devbox:stop && start`), when Ralph runs `gh pr view` or `git push`, then the existing token is reused (no re-auth required).

4. **Expired/revoked token surfaces clear pointer + Ralph pre-push gate halt-able.** Given an expired or revoked token, when `gh` invocations fail, then the failure surfaces a clear pointer to re-run `pnpm gh:auth` and Ralph's pre-push gate (Epic 3) treats this as a halt-able condition rather than silently retrying.

### OAuth flow mechanics (AC 1 + AC 2 + AC 3 + AC 4 anchor)

`gh`'s authentication (at apt-tracked `gh` version in Dockerfile:123-136) uses a device-code-style OAuth flow (default) or a PAT-based flow (`--with-token`):

1. `gh auth login` invoked with no existing credentials → prompts for host (default `github.com`), protocol (HTTPS/SSH), web-OAuth vs paste-token choice.
2. Operator selects web-OAuth → `gh` prints URL `https://github.com/login/device` + one-time code to stdout.
3. Operator reads URL + code from terminal (already surfaced via `docker exec -it` stdin/stdout connection).
4. Operator opens URL in host browser, completes GitHub OAuth, enters device code on the GitHub confirmation page.
5. `gh` polls GitHub's token endpoint until OAuth confirmation completes.
6. `gh` writes token entry under `/home/dev/.config/gh/hosts.yml` (default filename; substrate does NOT depend on the exact filename per SC-15).
7. `gh` proceeds to confirmation output ("Authenticated as @<username>") and exits 0.

Port-publication note: the device-code flow does NOT require a host-reachable callback server inside the container. GitHub's OAuth endpoint handles the code-exchange server-side. Compose's port list (lines 109-113: 3000 / 3001 / 6006 / 24679) does NOT need extension for Story 2.9. If GitHub pivots from device-code to PKCE-with-local-callback, that is a future amendment at that time (either publish a loopback port + update compose, or run a `socat` / `ssh -L`-adjacent port-forward from the host — both out of scope for 1.0).

Token persistence across restarts (AC 3): `/home/dev/.config/gh/` is under `/home/dev`, which is mounted from the `keel_home_dev` named volume (compose lines 90-92 + 170-171; Story 2.5 substrate). Named volumes survive container stop/start/restart; they are destroyed only on explicit `docker volume rm` or `pnpm devbox:clean --with-volumes`.

Re-auth on expiry (AC 4): when an expired/revoked token is loaded, `gh`'s own invocation failures (typically `error: authentication required` or HTTP 401 from GitHub API) surface to operator stdout via the same `docker exec -it` pipeline. Operator runs `pnpm gh:auth` again → OAuth flow repeats → new token overwrites old entry in `hosts.yml`. No wrapper-side state tracking required.

Ralph pre-push-gate halt-able handling (AC 4 second clause — Epic 3 scope): when Ralph's pre-push gate (Story 3.7 per architecture.md) runs `gh pr checks` or equivalent before pushing and detects an auth-broken gh response, it writes a halt sentinel `{"reason":"CI_BLOCKED","note":"gh not authed — run 'pnpm gh:auth'"}` and exits. The halt sentinel's `reason` field MUST be drawn from the PRD FR14k closed halt-reason enum (`EPIC_DONE`, `ALL_EPICS_DONE`, `AWAIT_MERGE`, `BUDGET_EXHAUSTED`, `CI_BLOCKED`, `SECURITY_CRITICAL`, `RALPH_STAGE_REGRESSION` — see `docs/invariants/ralph-execute.md` § Halt schema); `CI_BLOCKED` is the correct reason because an auth-broken gh blocks the CI-adjacent pre-push posture; silently retrying would waste iterations against a non-advancing contract. Story 2.9 does NOT deliver the halt-able handling itself — it pins the pointer-error surface (AC 4 clause 1) + the invariant of how Ralph MUST respond (AC 4 clause 2) so that Epic 3 Story 3.7 implements the halt-write without needing to re-invent the reason-taxonomy. This is the same "external contract described now for downstream consumer" pattern as Story 2.7's `KEEL_RALPH_MODE` env-var contract (Epic 3 consumer).
